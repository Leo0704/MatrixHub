# AI Native 内容矩阵系统 — 设计文档

> 日期：2026-03-26
> 状态：已确认，待实现
> MVP 范围：仅抖音平台

---

## 1. 项目定位

**一句话**：人说方向（推广什么产品），AI 完成一切（内容生成、分发、监控、迭代）。

**目标用户**：社交媒体运营团队、自媒体创作者、MCN 机构

**核心价值主张**：零配置启动，AI 驱动全流程，人只做决策。

---

## 2. 系统架构

```
用户（界面操作）
    │
    │  输入：产品链接/描述 + 内容类型 + 营销目标
    ▼
┌─────────────────────────────────────────────────────┐
│              核心驱动 AI（1个大模型）                  │
│   负责：决策 · 调度 · 内容策略制定 · 数据分析         │
└────────────────────────┬────────────────────────────┘
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  文案模型   │  │  图片模型   │  │  视频模型   │
│  (text)    │  │  (image)   │  │  (video)   │
└─────────────┘  └─────────────┘  └─────────────┘
        │
        ▼
┌─────────────┐
│  语音模型   │  ← TTS 配音（图文可选附加）
│  (voice)   │
└─────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────┐
│              抖音矩阵（用户登录的 N 个账号）           │
│   每个账号独立浏览器会话，独立内容，独立发布时间        │
└─────────────────────────────────────────────────────┘
```

---

## 3. 大模型配置

| 用途 | 模型数量 | 说明 |
|------|---------|------|
| 核心驱动 | 1 | 决策、调度、分析、内容策略制定 |
| 内容生成 | 4 | 文案、图片、视频、TTS 配音 |
| **合计** | **5** | |

所有模型由用户在设置页配置（BaseURL + API Key + 模型名）。

---

## 4. 账号管理

- 用户在账号管理页面自行登录维护（扫码、验证码等）
- 系统维护账号的登录会话状态
- **登录状态异常时**，系统通过界面通知用户
- 用户处理完毕后，系统继续执行
- MVP 仅支持抖音，多平台后续扩展

---

## 5. 推广执行流程

### 5.1 用户输入

| 字段 | 说明 |
|------|------|
| 产品信息 | 产品链接（AI 自动抓取）或 文字描述 |
| 内容类型 | 视频 / 图文 |
| 是否加语音 | 仅图文可选（勾选框） |
| 营销目标 | 曝光优先 / 互动优先 / 成交优先 |

### 5.2 AI 自动执行步骤

```
① 产品信息提取
   └─ 从链接抓取：产品名称、描述、卖点（抓不到的信息 AI 推断）

② 内容策略制定
   └─ 核心驱动 AI 分析产品，制定统一内容主题

③ 内容生成（每个账号独立）
   └─ 每个账号独立调用 4 个生成模型
   └─ 主题一致，表达各异（防平台识别为 spam）

④ 发布时间决策
   └─ 核心驱动 AI 根据账号历史活跃时间 + 平台流量峰值自主决定

⑤ 内容安全自审
   └─ 发布前 AI 自主检测违规词/敏感词
   └─ 发现问题 → AI 自主修改 → 再审 → 通过后发布

⑥ 发布执行
   └─ 每个账号独立内容独立发布
   └─ 失败 → 重试 3 次 → 跳过该账号 → 继续其他账号
   └─ 发布完成后汇总报告
```

### 5.3 48 小时监控周期

| 时间点 | 动作 |
|--------|------|
| 0.5 小时 | 爬取初始数据（验证发布成功） |
| 6 小时 | 爬取数据（算法是否开始推荐） |
| 24 小时 | 爬取第一天完整数据 |
| 48 小时 | 爬取第二天数据 → 生成效果报告 → 等待用户反馈 |

### 5.4 用户反馈循环

```
48小时报告
    │
    ├── [效果不错] → AI 同风格迭代继续发布（不限次数）
    │               每天自动生成简报，用户无需操作
    │
    └── [效果不好] → AI 换策略（风格/角度/配图）
                    自动执行 1 次迭代
                    继续 48 小时监控
                        │
                        ├── 第2次还不好 → AI 停止
                        │   通知你："建议换产品或人工介入"
                        │
                        └── 第2次变好 → 同风格继续，不限次数
```

## 6. 内容类型定义

| 用户选择 | AI 生成 |
|---------|--------|
| 视频 | 视频画面（可加 TTS 配音） |
| 图文 | 静态图片 + 文字（可加 TTS 配音） |

语音作为图文或视频的可选附加项，通过勾选控制。

---

## 7. 数据采集

### 7.1 可采集指标（爬虫可获取）

- 播放量 / 阅读量
- 点赞数
- 收藏数
- 评论数
- 转发数
- 粉丝变化

### 7.2 不可采集指标

- 完播率（平台不开放）
- ROI / 转化数据（无 API）

---

## 8. 通知机制

### 必须通知（界面弹窗）

- 账号被风控 / 封禁
- 内容违规被平台拦截
- 连续 2 次迭代失败

### 重要通知（界面弹窗）

- 发布全部完成
- 48 小时效果报告已生成
- 内容爆量（效果超预期）
- AI 自主决策换策略了

### 不通知

- 单个账号发布成功（用户可在 Dashboard 查看）
- 数据例行更新（报表随时可看）

---

## 9. 内容差异化策略

每个账号的内容独立生成，共用产品上下文信息，保证主题一致但表达各异。

同一产品在 20 个账号发布，生成 20 份不同的文案和图片。

---

## 10. 迭代决策权

| 操作 | AI 自主决定 | 必须通知用户 |
|------|-----------|------------|
| 文案风格变化 | ✅ | - |
| 配图风格变化 | ✅ | - |
| 发布时间微调 | ✅ | - |
| 内容角度变化 | ✅ | - |
| 更换核心营销卖点 | - | ✅ |
| 更换产品链接 | - | ✅ |
| 增加发布频率 | - | ✅ |
| 切换内容类型（视频↔图文） | - | ✅ |

---

## 11. 账号登录状态管理

- 登录态由用户维护（扫码、验证码、异地登录处理）
- 登录状态异常 → 系统通知用户
- 用户处理完毕 → 系统继续执行
- 账号被封 → 必须通知用户

## 12. 核心页面（MVP）

| 页面 | 职责 |
|------|------|
| 账号管理 | 抖音账号列表、登录状态、添加/删除账号 |
| 推广中心 | 输入产品 → 启动推广 → 查看进度 → 反馈效果 |
| 数据报表 | 各账号数据一览、效果对比、简报 |
| 设置 | 大模型配置（5个）、内容偏好设定 |

## 13. MVP 范围边界

### MVP 必须有
- 产品链接输入 → 内容生成 → 发布 → 数据采集 → 反馈迭代（核心闭环）
- 抖音账号列表 + 登录状态监控
- 简报（播放/互动数据表格）

### MVP 去掉（后期加）
- 语音作为独立内容形式
- 账号分组/标签功能
- 历史推广记录独立页面
- 日报推送通知
- 竞品监控
- 多产品并行

## 14. 关键矛盾点（当前代码 vs 设计）

| 当前代码问题 | 设计要求 |
|------------|---------|
| Service Worker 循环从未被调用 | 必须修复，后台任务执行是核心 |
| automation:confirm 架构反向 | 取消所有人工确认节点 |
| AI Director 只做失败分析 | 升级为全流程决策中枢 |
| Pipeline 需要用户手动配置 | 改为 AI 自主决策参数 |
| TaskType 定义三处不一致 | 统一为单一数据源 |
| page_agent 类型 DB 中不存在 | 设计中不使用 page_agent |

## 15. 产品信息抓取

### 15.1 抓取字段

| 字段 | 是否必须 | 说明 |
|------|---------|------|
| 产品名称 | ✅ 必须 | 主标题 |
| 产品描述 | ✅ 必须 | 核心卖点、功能 |
| 价格 | 可选 | 抓不到则标注"未知" |
| 规格/参数 | 可选 | 抓不到则标注"未知" |
| 品牌 | 可选 | 抓不到则 AI 推断 |
| 适用人群 | 可选 | 抓不到则 AI 推断 |
| 产品图片 | ✅ 必须 | 用于生成素材 |

### 15.2 抓取失败处理

- 抓不到 → **提示用户手动填写**
- 用户输入：链接（必填）+ 补充文字说明（可选）
- 用户可选择"手动填写"绕过链接抓取

### 15.3 技术要求

- 必须有 SSRF 防护（拒绝内网地址）
- 支持主流电商/产品页面解析

---

## 16. 内容违规检测

### 16.1 检测范围

| 类型 | 说明 |
|------|------|
| 极限用语 | 平台禁止的"最"字辈词汇库 |
| 虚假宣传 | 绝对化功效声称（7天、永久、100%等） |
| 敏感行业 | 医疗/药品/金融/彩票/烟草 |
| 违禁品 | 平台禁售品类 |
| 版权侵权 | 未经授权的品牌 logo、产品图、明星素材 |

### 16.2 处理流程

检测到违规 → AI 自主修改文案 → 再审 → 通过后发布

## 17. 简报内容模板

### 48 小时效果简报

| 内容 | 说明 |
|------|------|
| 各账号播放量/阅读量 | 表格展示 |
| 各账号互动数据 | 点赞/评论/收藏 |
| 账号健康状态 | 是否被限流 |
| 效果最好账号 Top3 | 按综合表现排名 |
| 效果最差账号 | 标注原因（可选） |
| 建议下一步 | 继续/换策略/停止 |

## 18. 用户反馈 UI

- 位置：48h 简报页面底部内嵌
- 交互：简报生成后，页面底部显示"效果好" / "效果不好"两个按钮
- 路径：简报 → 反馈 → AI 立即执行，操作路径最短
